날짜: 2026-07-12

- Fresh Windows runner에서 upload integrity, Electron package와 release gate를 검증했습니다.
- 검증된 네 자산을 prerelease로 게시한 뒤 exact tag published updater를 실제 설치본에
  적용하고 stable/latest로 승격했습니다.

  만들지 않았습니다.
- 기존 `v0.4.15 -> v0.4.16` published updater 증거는
  `2026-05-12-deployment-auth-evidence-handoff.md`에 그대로 보존합니다.

## Release 후 artifact privacy 교정

Release 후 16개 evidence JSON을 새 privacy scanner로 다시 검사했습니다. Package 12개,
browser 1개와 published-updater JSON 1개는 통과했지만 published-updater JSON 2개는
다음 경로와 위반 label 때문에 실패했습니다.

- `$.payload.latestReleaseUrl`
- child process projection의 `stdoutTail`, `stderrTail`
- URL, smoke temp path, terminal escape

재감사에서 token이나 credential은 발견되지 않았습니다. 따라서 이 handoff의 package/upload,
실제 `v0.4.16 -> v0.4.20` updater apply와 stable asset 증거는 유효하지만,
`smoke-windows-published-update-v0.4.20`은 privacy-safe evidence로 사용하지 않습니다.
Sanitizer와 upload 전 fail-closed scanner를 적용한 `v0.4.21` evidence가 이 범위를 대체합니다.

## Upload integrity 증거

Fresh Windows package gate에서 다음 항목이 모두 통과했습니다.

- Exact-limit artifact의 byte size와 SHA-256이 request body와 일치했습니다.
- Stage와 final이 같은 directory에 있었고 no-replace hard-link publish가 성공했습니다.
- Reserved stage를 실행 중 실제로 관찰한 뒤 abort했고 동일 path가 unlink됐습니다.
- Age floor를 넘긴 reserved stage는 cleanup됐습니다.
- 정상 committed `*.part` artifact는 reserved-stage cleanup 뒤에도 유지됐습니다.
- 두 번째 instance에 `CODEXMUX_UPLOADS_DISABLED=1`을 적용하자 exact upload route만 `503`을
  반환했고 health, auth와 기존 artifact 상태는 보존됐습니다.

이 결과로 Linux dev/prod 증거까지만 있던 ADR-027의 Windows hard-link/delete/package 경계가
충족됐습니다.

## Rollback과 제한

- 이 release는 unsigned 내부 배포 build입니다. Public code-signing trust와 SmartScreen
  reputation은 검증 범위가 아니며 내부 신뢰 절차와 installer 경고가 필요합니다.
- macOS package와 npm publish는 Windows stable gate에서 제외했습니다. 이 handoff는 해당
  surface의 배포 성공을 의미하지 않습니다.
- Upload 장애 시 `CODEXMUX_UPLOADS_DISABLED=1`로 재시작해 두 upload route만 `503`으로
  격리합니다. Health, auth와 기존 artifact는 유지한 채 patched build로 forward recovery합니다.
- Updater는 자동 downgrade를 제공하지 않으며 `v0.4.20` 상태에서 `v0.4.16`으로 되돌리는
  호환성도 검증하지 않았습니다. 문제가 있으면 upload kill switch나 영향 surface 격리 뒤
  forward fix를 새 version으로 release하고 gate를 반복합니다.
- 다음 stable release는 workflow의 baseline tag, version과 installer SHA-256을 당시 직전
  stable로 다시 pin해야 합니다.
