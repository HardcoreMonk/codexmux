# My Agent — 자율 코딩 에이전트

## 배경

현재 여러 프로젝트의 할일을 사용자가 직접 인지하고, 하나씩 터미널에서 Claude Code를 실행하고, 결과를 확인하고 있다.
이 과정을 에이전트에게 위임하여, 지시만 내리면 계획 수립 → 실행 → 보고까지 자율적으로 수행하게 한다.

## 핵심 컨셉

**인턴 모델** — 에이전트는 인턴처럼 동작한다:
- 할일을 받으면 혼자 계획 짜고 실행
- 막히면 찾아와서 질문
- 중간중간 보고
- 끝나면 결과 보고
- 과거에 함께한 작업을 기억

**멀티 에이전트** — 에이전트는 여러 개 생성/운영할 수 있다:
- 역할별 분리 (백엔드 전담, 프론트 전담, 리뷰어 등)
- 프로젝트별 분리 (Project A 전담, Project B 전담)
- 각 에이전트가 독립적으로 미션을 수행
- 같은 프로젝트에 여러 에이전트가 붙을 수 있음

## 실행 방식

**사람이 purplemux를 쓰는 방식과 완전히 동일하게 에이전트가 동작한다.**

### 구조

```
에이전트 A 워크스페이스 (계획/관리 전용)
├── 에이전트 A가 스스로 계획 수립, 태스크 관리, 판단하는 공간

에이전트 B 워크스페이스 (계획/관리 전용)
├── 에이전트 B가 스스로 계획 수립, 태스크 관리, 판단하는 공간

프로젝트 A 워크스페이스
├── [에이전트 A] Task 1 탭 → Claude Code 세션
├── [에이전트 A] Task 2 탭 → Claude Code 세션
└── [에이전트 B] Task 3 탭 → Claude Code 세션  ← 다른 에이전트도 가능

프로젝트 B 워크스페이스
├── [에이전트 B] Task 1 탭 → Claude Code 세션
└── ...
```

### 흐름

1. 에이전트가 자기 워크스페이스에서 **Mission을 분석하고 계획 수립**
2. 해당 프로젝트의 **워크스페이스에 탭을 생성**
3. 탭에서 **Claude Code 대화 세션을 시작**
4. Step을 하나씩 **대화를 주고받으며** 처리 (원샷이 아닌 인터랙티브 세션)
5. 사용자는 아무 탭이나 열어 **진행 상황을 실시간으로 관찰** 가능
6. 병렬 태스크 시 탭을 여러 개 생성

### 제어 방식

에이전트는 UI 레이어를 거치지 않고 **tmux를 직접 제어**한다:
- 탭 생성 → tmux API로 세션/윈도우 생성
- Claude Code 실행/입력 → tmux send-keys
- 상태 확인 → 기존 상태 감지 로직 재사용 (pane title, 세션 감지)
- 결과 확인 → jsonl 파싱 또는 tmux 버퍼

UI(워크스페이스, 탭, Claude Code 패널, 타임라인)는 **사용자가 관찰하는 용도**로만 사용.
기존 purplemux 인프라를 활용하되, 에이전트 전용으로 더 효율적인 경로가 있으면 그쪽을 선택.

- CLAUDE.md를 통해 프로젝트별 시스템 프롬프트 적용
- Claude API 대신 Claude Code 사용 (비용 효율)

## 태스크 모델

3단계 트리 구조. 지시의 복잡도에 따라 자연스럽게 스케일링된다.

```
Mission (최상위 지시)
└── Task (실행 단위)
    └── Step (개별 동작)
```

### 상태

```
pending → running → completed
                  → blocked (사용자 확인 필요)
                  → failed
```

- `blocked` 상태가 되면 해당 태스크만 멈추고, 다른 태스크는 계속 진행
- 사용자에게 알림 전송

### 롤링 계획

- Mission을 받으면 에이전트가 Task 목록을 생성
- 각 Task 시작 시 Step 목록을 구체화
- 처음부터 전체를 확정하지 않고, 앞 단계 결과를 반영하여 뒷 단계를 조정

### 예시 — 간단한 일

```
Mission: "README 오타 수정해줘"
└── Task 1: 오타 수정
    └── Step 1: 수정 후 커밋 ✅
```

### 예시 — 복잡한 일

```
Mission: "A 프로젝트 인증 시스템 교체"
├── Task 1: 현행 분석
│   ├── Step 1: 기존 코드 분석 ✅
│   ├── Step 2: 의존성 파악 ✅
│   └── Step 3: 영향 범위 정리 ✅
├── Task 2: 설계
│   ├── Step 1: 방식 비교 → blocked 🔴 "JWT vs Session?"
│   └── Step 2: API 스펙 작성 (대기)
├── Task 3: 구현 (대기)
├── Task 4: 테스트 (대기)
└── Task 5: 마이그레이션 (대기)
```

## 커뮤니케이션

- 채팅 UI를 통해 사용자와 대화
- **에이전트가 먼저 말 걸 수 있음** (WebSocket push)
  - 질문, 중간보고, 완료보고, 승인 요청
- 사용자가 응답하지 않으면 해당 태스크만 blocked, 나머지는 계속 진행
- 메시지 유형 구분으로 중요도 필터링 가능

## 메모리 시스템

파일 기반 영속 메모리. 계층 구조로 관리하되, 느슨하게 연결하여 확장 가능하게 한다.

### 구조

```
~/.purplemux/agents/
├── shared/                            # 전체 에이전트 공유
│   ├── user.md                        # 사용자 선호도, 작업 스타일
│   ├── global-learnings.md            # 공통 교훈
│   └── projects/                      # 프로젝트별 공유 지식
│       ├── project-a/
│       │   ├── context.md             # 기술 스택, 컨벤션, 아키텍처
│       │   ├── learnings.md           # 이 프로젝트에서 배운 것
│       │   └── index.md              # 주요 파일/모듈 맵 (캐싱)
│       └── project-b/
│           └── ...
├── backend-bot/                       # 에이전트별 독립 공간
│   ├── config.md                      # 역할, 담당 프로젝트, 자율성 수준
│   ├── memory/                        # 이 에이전트만의 경험/메모
│   └── missions/                      # 이 에이전트의 미션 이력
├── frontend-bot/
│   ├── config.md
│   ├── memory/
│   └── missions/
└── ...
```

### 메모리 계층

| 계층 | 범위 | 내용 | 수명 |
|---|---|---|---|
| shared/user.md | 전체 | "커밋 메시지 한글", "PR 크게 묶는 거 선호" | 반영구 |
| shared/global-learnings.md | 전체 | 프로젝트 공통 패턴, API 주의사항 | 반영구 |
| shared/projects/*/context.md | 프로젝트 | 기술 스택, 폴더 구조, 빌드 방법 | 프로젝트 수명 |
| shared/projects/*/learnings.md | 프로젝트 | "이 프로젝트 CI가 느려서 로컬 테스트 필수" | 누적 |
| shared/projects/*/index.md | 프로젝트 | 주요 진입점, 모듈 관계, 의존성 맵 | 변경 시 갱신 |
| {agent}/memory/ | 에이전트 | 이 에이전트만의 경험, 판단 기록 | 누적 |
| {agent}/missions/ | 에이전트 | 완료된 미션 이력, 결과 요약 | 아카이브 |

### 크로스 학습

- 프로젝트 메모리(shared/projects/)는 **모든 에이전트가 읽고 쓸 수 있음**
- 에이전트 A가 배운 교훈이 범용적이면 → shared로 승격
- 에이전트별 memory/는 해당 에이전트만의 맥락 (다른 에이전트가 참조는 가능)
- 인덱스 정보는 git diff 기반으로 변경 감지 시 갱신

### 설계 원칙

- **파일 = 메모리 단위**: 각 파일이 독립적으로 읽고 쓸 수 있는 단위
- **느슨한 연결**: 디렉토리 컨벤션으로만 연결, 별도 DB나 스키마 없음
- **점진적 축적**: 처음엔 거의 비어있고, 작업하면서 자연스럽게 채워짐
- **에이전트가 스스로 관리**: 메모리 작성/갱신/정리를 에이전트 자신이 수행
- **에이전트 추가 = 디렉토리 추가**: 새 에이전트는 폴더 하나 만들면 끝

## 해결해야 할 문제

### 1. claude -p 출력 파싱

- 자연어 출력을 구조화된 데이터로 변환해야 함
- CLAUDE.md에 JSON 응답 규칙을 넣어 파싱 가능하게 유도
- exit code, 에러 패턴 기반 실패 감지

### 2. 자율성 경계

- 언제 알아서 하고 언제 물어볼지의 기준
- 초반엔 보수적으로 (많이 물어봄)
- 메모리에 "이런 건 알아서 해도 됨" 축적하며 자율성 확대

### 3. 컨텍스트 유실

- 긴 태스크에서 앞 단계 결과를 까먹을 수 있음
- 각 Step 결과를 파일로 저장
- 다음 Step 실행 시 이전 결과 파일을 컨텍스트로 전달

### 4. 비동기 대기

- 사용자 응답을 기다리는 동안 다른 태스크는 계속 진행
- 태스크별 독립 실행 (세션 분리)
