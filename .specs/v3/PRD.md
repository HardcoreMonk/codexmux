# v3 요구사항 정리

## 출처

- `.specs/v3/requirements/overview.md` — 프로젝트 개요, 완료 사항, 기술 스택, 로드맵
- `.specs/v3/requirements/phase3-surface.md` — Phase 3 Surface(탭) 상세 요구사항

## 프로젝트 비전

웹 기반 영속적 작업 환경. 로컬 PC에 서버를 띄우고, 브라우저에서 터미널 + Claude Code를 통합 관리하는 도구.

**핵심 가치**: 한번 열어둔 작업이 서버 재시작 후에도 그 자리에 그대로 있는 것.

## 완료 사항

| 항목 | 상태 |
|---|---|
| Phase 1: 웹 터미널 (xterm.js + node-pty + WebSocket) | ✅ 완료 |
| Custom Server 전환 (API Route → server.ts) | ✅ 완료 |
| Phase 2: tmux 백엔드 (세션 영속성, detaching 플래그, close code 정책) | ✅ 완료 |

## v3 목표

**Phase 3 — Surface(탭)**: 단일 터미널을 탭 기반 다중 터미널로 확장한다. 각 탭은 독립된 tmux 세션과 1:1 매핑되며, 탭 생성/전환/삭제/순서 변경/이름 변경이 가능하다.

**완료 조건**: 탭으로 여러 터미널을 관리할 수 있다. 탭 목록은 서버 재시작/새로고침 후에도 복원된다.

## 주요 요구사항

### 메인 페이지 (`/`)

#### 탭 바 UI

- 화면 상단에 탭 바, 나머지 영역에 활성 탭의 터미널을 렌더링
- 각 탭에 이름(기본: "Terminal 1", "Terminal 2", ...)과 닫기 버튼(×) 표시
- 탭 바 우측에 새 탭 추가 버튼(+) 배치
- 활성 탭은 시각적으로 구분 (배경색, 하단 보더 등 — Muted 팔레트 준수)
- 탭이 1개일 때도 탭 바 표시 (일관된 UI)
- 탭 개수가 많아지면 탭 바 가로 스크롤 또는 축소
- 탭 바 높이를 최소한으로 유지하여 터미널 영역 침범 최소화

#### 탭 생성

- + 버튼 클릭으로 새 탭 생성
- 새 탭 생성 시 서버에 새 tmux 세션 생성 요청 → WebSocket 연결 (`/api/terminal`, 세션 파라미터 없음)
- 새 탭은 맨 우측에 추가되고 자동으로 활성화
- 기본 이름: "Terminal {N}" (생성 순서 기반)

#### 탭 전환

- 탭 클릭 시 해당 탭의 터미널로 전환
- 전환 시 현재 WebSocket을 닫고(detach), 새 탭의 세션 ID로 새 WebSocket 연결 (`/api/terminal?session={id}`)
- 단일 xterm.js 인스턴스 재활용 — 전환 시 화면 초기화 후 tmux redraw 수신
- 비활성 탭의 tmux 세션은 백그라운드에서 계속 실행

#### 탭 삭제

- 닫기 버튼(×) 클릭으로 탭 삭제
- 삭제 시 tmux 세션 `kill-session` 종료
- 활성 탭 삭제 시 인접 탭으로 자동 전환 (우측 우선, 없으면 좌측)
- 마지막 탭 삭제 시 새 탭 자동 생성 (빈 상태 방지)
- 탭 내 `exit` 실행 시에도 해당 탭이 닫힘 (Phase 2 세션 종료 정책 확장)

#### 탭 순서 변경

- 탭 드래그 앤 드롭으로 순서 변경
- 변경된 순서는 `tabs.json`에 즉시 반영

#### 탭 이름 변경

- 탭 더블클릭으로 인라인 편집 모드 진입
- Enter 또는 blur 시 이름 확정 → `tabs.json`에 저장

### 서버 — WebSocket (`/api/terminal`)

#### 세션 ID 파라미터

- URL 쿼리 파라미터로 세션 ID를 전달한다: `/api/terminal?session=pt-a1b2c3-d4e5f6-g7h8i9`
- `session` 파라미터 있음 → 해당 tmux 세션에 attach
- `session` 파라미터 없음 → 새 tmux 세션 생성 + attach (Phase 2 하위 호환)
- `server.ts` upgrade 핸들러에서 `request.url` 파싱
- 기존 바이너리 메시지 프로토콜(0x00~0x03) 변경 없음

#### Phase 2 정책 유지

- detaching 플래그, close code 정책 (1000/1001/1011/1013) 그대로 유지
- 탭 전환 시 WebSocket close → 서버: `detaching = true` → tmux detach → 세션 유지

### 서버 — 탭 관리 API (`/api/tabs`, 신규)

#### 탭 목록 조회

- 클라이언트 페이지 로드 시 탭 목록을 조회한다
- 서버는 `~/.purple-terminal/tabs.json` + tmux 세션 목록을 크로스 체크하여 반환
  - tabs.json에 있지만 tmux 세션이 없는 탭 → 제거
  - tmux 세션이 있지만 tabs.json에 없는 세션 → 탭으로 추가 (orphan 복구)
- 응답: 탭 목록 (탭 ID, 세션 이름, 탭 이름, 순서, 활성 탭)

#### 탭 상태 저장

- 탭 생성/삭제/이름 변경/순서 변경 시 `tabs.json`에 자동 저장
- 디바운스 적용으로 빈번한 디스크 I/O 방지
- 활성 탭 정보도 함께 저장 (새로고침 시 복원)

### 탭 영속성

- **서버 재시작**: `tabs.json` + tmux 세션 크로스 체크 → 탭 목록 복원 → 클라이언트 재연결 시 활성 탭 자동 연결
- **새로고침**: 페이지 로드 시 `/api/tabs`로 탭 목록 조회 → 마지막 활성 탭의 세션에 WebSocket 연결
- **탭 내 exit**: 해당 탭 제거 + tabs.json 갱신 → 인접 탭 전환

## 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 탭 전환 속도 | WebSocket 재연결 + tmux attach가 사용자에게 지연으로 체감되지 않아야 함 |
| 백그라운드 세션 | 비활성 탭의 프로세스(빌드, 테스트 등)가 중단되지 않아야 함 |
| 탭 바 최소 침범 | 탭 바 높이를 최소한으로 유지, Muted 팔레트 다크 테마 적용 |
| Phase 2 호환 | 세션 영속성, detaching 플래그, close code 정책이 탭 환경에서 동일하게 동작 |
| 프로토콜 호환 | 기존 바이너리 프로토콜(0x00~0x03) 변경 없음 |

## 기술 구성

```
Browser                              Server (Custom)                  tmux (-L purple)
┌──────────────────────────┐          ┌─────────────────┐              ┌──────────────┐
│ ┌─Tab 1─┬─Tab 2─┬─ + ─┐ │          │  server.ts      │              │ pt-...-...-A │
│ │                      │ │   WS     │  (세션 매칭)     │   attach     │ pt-...-...-B │
│ │   활성 탭 터미널       │ │ ◄──────► │                 │ ◄──────────► │ pt-...-...-C │
│ │   (단일 xterm.js)    │ │          │  /api/tabs      │              └──────────────┘
│ └──────────────────────┘ │   HTTP   │  (탭 목록 관리)  │
│                          │ ◄──────► │                 │   tabs.json
└──────────────────────────┘          └─────────────────┘   ~/.purple-terminal/
```

| 항목 | Phase 2 | Phase 3 |
|---|---|---|
| tmux 세션 수 | 1개 | 탭 수만큼 N개 |
| 세션 매칭 | 첫 번째 `pt-*` 세션 | URL 파라미터 `?session={id}` |
| WebSocket | 단일 연결 | 단일 연결, 탭 전환 시 끊고 재연결 |
| xterm.js | 단일 인스턴스 | 단일 인스턴스 재활용 |
| UI | 전체 화면 터미널 | 탭 바 + 터미널 영역 |
| 탭 상태 | 없음 | `~/.purple-terminal/tabs.json` |

## 검증 시나리오

1. **탭 생성**: + 버튼 클릭 시 새 탭이 생성되고 새 터미널이 시작된다
2. **탭 전환**: 탭 클릭 시 해당 터미널로 전환된다. 이전 탭의 프로세스는 계속 실행된다
3. **탭 삭제**: × 버튼 클릭 시 탭이 닫히고 tmux 세션이 종료된다
4. **마지막 탭 삭제**: 마지막 탭을 닫으면 새 탭이 자동 생성된다
5. **탭 내 exit**: 터미널에서 exit 실행 시 해당 탭이 닫힌다
6. **서버 재시작 복원**: 서버 재시작 후 모든 탭과 터미널 상태가 복원된다
7. **새로고침 복원**: 브라우저 새로고침 후 탭 목록과 활성 탭이 복원된다
8. **백그라운드 실행**: Tab 1에서 빌드 실행 중 Tab 2로 전환 → Tab 1로 돌아오면 빌드가 계속 진행 중이다
9. **다중 탭 동시 프로세스**: 여러 탭에서 각각 다른 작업 실행 중 서버를 재시작해도 모두 복원된다
10. **탭 바 UI**: 탭이 많아져도 UI가 깨지지 않는다 (스크롤 또는 축소)
11. **탭 순서 변경**: 탭 드래그로 순서를 변경하면 재시작 후에도 순서가 유지된다
12. **탭 이름 변경**: 탭 더블클릭으로 이름을 변경하면 재시작 후에도 이름이 유지된다

## 제약 조건 / 참고 사항

- **단일 WebSocket 연결**: 탭 전환 시 기존 WebSocket을 닫고 새로 연결. 다중 WebSocket 동시 유지는 하지 않음. 전환 속도가 체감 지연 없도록 WebSocket 연결 + tmux attach가 빠르게 완료되어야 함
- **단일 xterm.js 인스턴스**: 탭마다 별도 인스턴스를 생성하지 않고 하나를 재활용. 전환 시 화면 초기화 후 tmux redraw로 복원. 메모리 효율적이지만 전환 시 잠깐 빈 화면이 보일 수 있음
- **tabs.json 저장**: `~/.purple-terminal/tabs.json`에 탭 상태 저장. 디바운스 적용으로 빈번한 쓰기 방지. 파일이 없거나 파싱 실패 시 빈 탭 목록으로 시작
- **orphan 세션 복구**: tmux에 세션이 있지만 tabs.json에 없는 경우 자동으로 탭으로 추가. 데이터 손실 방지
- **탭 전환 시 detaching**: 탭 전환은 "의도적 detach"이므로 Phase 2의 `detaching` 플래그가 동작. 전환 시 session-ended UI가 표시되면 안 됨
- **탭 바 디자인**: Muted 팔레트 다크 테마 적용. `docs/STYLE.md` 준수. 높이를 최소한으로 (28~32px 수준) 유지
- **Phase 2 하위 호환**: `?session` 파라미터 없이 WebSocket 연결하면 Phase 2와 동일하게 동작 (첫 번째 pt-* 세션 또는 새 세션)

## 확정된 결정사항

| 항목 | 결정 | 근거 |
|---|---|---|
| WebSocket 연결 전략 | 단일 연결, 탭 전환 시 끊고 재연결 | 리소스 효율적, 서버 부하 최소화 |
| 탭 상태 저장 위치 | 서버 `~/.purple-terminal/tabs.json` | 다른 브라우저에서도 복원 가능 |
| xterm.js 인스턴스 관리 | 단일 인스턴스 재활용 | 메모리 절약, tmux redraw로 화면 복원 |
| 탭 순서/이름 변경 | Phase 3 범위에 포함 | 드래그 재정렬 + 더블클릭 이름 편집 |
| 세션 ID 전달 방식 | URL 쿼리 파라미터 (`?session={id}`) | 프로토콜 변경 없음, Phase 2 하위 호환, 디버깅 용이 |
