# v12 요구사항 정리

## 출처

- `.specs/v12/requirements/overview.md` — 프로젝트 개요 및 로드맵
- `.specs/v12/requirements/phase12-usage-statistics.md` — Phase 12 사용량 통계 페이지 PRD

## 페이지 목록 (도출)

v12는 기존 터미널 UI와 별도로 독립적인 통계 대시보드 페이지를 추가한다.

| 페이지 | 설명 | 우선순위 |
|---|---|---|
| 통계 대시보드 | 사용량 통계를 차트와 지표로 시각화하는 전용 페이지 | P0 |
| 통계 데이터 API | 4종 데이터 소스를 파싱하여 통계 데이터를 제공하는 서버 API | P0 |

## 주요 요구사항

### 통계 페이지 라우팅 및 네비게이션

- 사이드바 하단에 통계 진입 메뉴 추가 (lucide-react `BarChart3` 아이콘)
- 클릭 시 전용 통계 페이지로 전환 — 기존 Workspace 터미널 레이아웃과 완전히 분리된 뷰
- Workspace 클릭으로 터미널 레이아웃 복귀 (통계 페이지 언마운트)
- 통계 페이지는 Next.js Pages Router의 별도 라우트 또는 동일 페이지 내 뷰 전환으로 구현

### 데이터 소스 파싱 (서버)

4종의 Claude Code 로컬 파일에서 통계 데이터를 수집한다:

| 파일 | 경로 | 역할 | 로딩 단계 |
|---|---|---|---|
| stats-cache | `~/.claude/stats-cache.json` | 일별 집계, 모델별 토큰, 시간대별 분포 | 1단계 (즉시) |
| 세션 JSONL | `~/.claude/projects/*/*.jsonl` | 세션별 메시지 원본, 토큰 usage | 2단계 (백그라운드) |
| facets | `~/.claude/usage-data/facets/*.json` | 세션 카테고리, 목표 달성도, 만족도 | 3단계 (백그라운드) |
| history | `~/.claude/history.jsonl` | 전체 사용자 입력 히스토리 | 3단계 (백그라운드) |

- **2단계 로딩**: stats-cache를 먼저 로드하여 개요/차트를 즉시 렌더링, JSONL/facets는 백그라운드에서 파싱
- stats-cache.json의 실제 구조를 분석하여 어떤 데이터를 즉시 활용 가능한지 확인 필요

### 개요 섹션

| 지표 | 데이터 소스 | 표시 형태 |
|---|---|---|
| 총 세션 수 | stats-cache 또는 프로젝트 디렉토리 스캔 | 숫자 카드 |
| 총 메시지 수 | stats-cache | 숫자 카드 |
| 오늘/이번 주/이번 달 사용량 | stats-cache (일별 집계) | 숫자 카드 |
| 누적 추이 | stats-cache (일별 집계) | 라인 차트 (`--chart-1`) |

### 토큰 사용량 섹션

- **모델별 토큰**: 입력/출력/캐시 분리, 바 차트
  - Opus → `var(--chart-5)` (ui-purple), Sonnet → `var(--chart-1)` (ui-blue), Haiku → `var(--chart-2)` (ui-teal)
- **일별 토큰 추이**: 에어리어 차트, 입력/출력 스택
  - 입력 → `var(--chart-1)`, 출력 → `var(--chart-2)`
- **모델별 비율**: 도넛 차트
  - 각 모델에 의미 기반 색상 배정 (`docs/STYLE.md` 지침)

### 활동 패턴 섹션

- **시간대별 히트맵**: GitHub 잔디 스타일, 주 단위 × 시간 단위 그리드
  - 색상: `var(--chart-2)` (ui-teal) 계열 단일 색조, 강도로 빈도 표현
- **일별 차트**: 메시지 수 / 세션 수 / 도구 호출 수 — 멀티 시리즈 바 차트
- **요일별 패턴**: 요일별 평균 사용량 비교

### 프로젝트별 분석 섹션

- Workspace별 세션 수 / 메시지 수 — 수평 바 차트
- 프로젝트별 토큰 소비 비교 — 수평 바 차트
- 프로젝트 이름: Workspace 디렉토리 경로의 마지막 세그먼트

### 세션 분석 섹션

- 평균 세션 길이 / 최장 세션 — 숫자 카드
- 세션 카테고리 분포 — facets 데이터, 도넛 차트
- 목표 달성도 통계 — facets 데이터, 바 차트 또는 게이지

### 기간 필터

- 상단 고정: `오늘 | 7일 | 30일 | 전체` 탭 또는 토글 버튼
- 기본값: 30일
- 필터 변경 시 모든 섹션의 데이터와 차트가 해당 기간으로 갱신
- 필터 변경 시 stats-cache에서 필터링 가능한 데이터는 클라이언트 측 필터, 그 외는 서버 재조회

## 제약 조건 / 참고 사항

### 기술적 제약

- **stats-cache.json 구조 의존**: Claude Code가 저장하는 파일이므로 버전에 따라 구조가 변경될 수 있음. 파싱 시 defensive하게 처리
- **JSONL 대용량**: 장기 사용자의 세션 파일은 수백 MB 가능. 서버 측 스트리밍 파싱 + 결과만 클라이언트 전송
- **facets 데이터 가용성**: 모든 세션에 facets가 있는지 확인 필요. 없는 경우 해당 섹션 빈 상태 처리
- **차트 색상**: `--chart-1` ~ `--chart-9` CSS 변수 사용 필수, `docs/STYLE.md` 지침에 따라 의미 기반 색상 선택

### UX 고려사항

- **점진적 렌더링**: stats-cache 기반 섹션은 즉시 표시, JSONL 기반 섹션은 skeleton → 점진적 채워짐
- **빈 상태**: 데이터가 없는 섹션은 "데이터가 없습니다" 안내 + 기간 변경 제안
- **차트 인터랙션**: 호버 시 수치 Tooltip, 차트 영역 클릭/드래그로 기간 줌인은 범위 외
- **반응형**: 통계 페이지는 스크롤 가능한 단일 컬럼 레이아웃, 차트는 컨테이너 너비에 반응

### 성능

- stats-cache.json 로드 → 개요 렌더링: **1초 이내**
- JSONL 파싱은 서버 API 응답 (REST), 클라이언트 스트리밍 파싱 아님
- 차트 데이터 포인트: 365개(일별 1년) 기준 부드러운 렌더링
- 서버 측 집계 후 요약 데이터만 전송 (원본 엔트리 전체를 보내지 않음)

## 확정 사항 (미확인 → 해결)

- [x] `stats-cache.json` 구조 → 현재 버전 기준으로 파악하여 구현. 구조 변경에 대비해 각 섹션을 Error Boundary로 감싸서 파싱 실패 시 해당 섹션만 에러 표시 (전체 페이지 크래시 방지)
- [x] `facets/*.json` 구조 → 동일하게 현재 버전 기준 파악 + Error Boundary로 안전하게 처리
- [x] `history.jsonl` 활용 → 부하가 크지 않다면 다양한 지표 제공 (자주 사용하는 명령어, 입력 길이 분포, 시간대별 입력 패턴 등). 파일 크기가 큰 경우 서버 측 스트리밍 파싱
- [x] 라우팅 → 별도 Next.js 라우트 (`/stats`)로 구현. 기존 터미널 WebSocket 연결은 끊김 (통계 페이지에서 터미널 불필요). Workspace 복귀 시 재연결
